# 9편 — 두 번째 품질 패스: 경고 0과 거대 페이지 분해

기능 확장(7편)과 회고(8편)를 끝내고 결제 단계로 넘어가기 전에, 한 번 더 품질 패스를
돌렸다. 6편이 "새던 데이터와 lint 게이트 도입"이었다면 이번은 그 게이트를 실제로
조이는 작업이다. 경고 310건을 0으로 만들고 룰을 전부 error로 올렸으며, 500줄대
페이지 컴포넌트들을 분해했다. 화면이 바뀐 작업이 아니라서 이번 편은 스크린샷이 없다.

## 0. 머지 전 소동: "1 secret uncovered!"

전 세션까지의 작업이 쌓인 PR을 머지하려는데 GitGuardian 체크가 빨간불이었다.

```
1 secret uncovered!
#### 1 secret were uncovered from the scan of 33 commits in your pull request.
```

세부 내역은 대시보드에서만 보여서, 브랜치 33커밋 전체를 직접 스캔했다. Anthropic 키
패턴, JWT 형태 문자열, 자격증명 대입문을 히스토리 단위로 훑었는데 걸리는 건 전부
`.env.example`의 플레이스홀더였다 — `POSTGRES_PASSWORD=postgres`와
`JWT_SECRET=change-me`. 실제 API 키로는 히스토리 검색(`git log -S`)에서 단 한 건도
나오지 않았고, 진짜 키가 든 `.env` 계열 파일은 gitignore가 막고 있었다. 결론은
플레이스홀더를 일반 비밀번호로 오인한 오탐. 안심하고 머지했다.

교훈이라면, 예시 파일의 자격증명 칸도 `postgres` 같은 그럴듯한 값 대신 한눈에 가짜인
값을 쓰면 스캐너와 입씨름할 일이 줄어든다는 것.

## 1. 경고 310건의 분포

lint 게이트를 도입할 때(6편) 기존 위반이 많은 type-safety 계열 룰을 warn으로 깔아뒀다.
그 빚이 얼마나 쌓였는지 세어 보니 모노레포 전체 310건이었다.

| 워크스페이스 | 경고 | 주범 |
|---|---|---|
| 클라이언트 | 195 | 탈윈드 클래스 카테고리 정렬 157 |
| 서버 | 111 | 테스트의 any 응답 접근 약 80 |
| pokedex-core | 4 | 잡다 |

## 2. 탈윈드 정렬 경고는 --fix로 안 잡힌다

157건이니 당연히 자동 수정부터 시도했다. 그런데 `eslint --fix`를 돌려도 경고 수가
그대로였다. 룰 메타에는 분명 `fixable: 'code'`가 있는데.

플러그인 소스를 열어 보고 이유를 알았다. 이 룰은 경고를 두 종류 낸다. 하나는 cn()
인자들의 카테고리 순서가 어긋났다는 것(이건 fixer가 인자를 재정렬해 고친다), 다른
하나는 **한 문자열 안에 서로 다른 카테고리의 클래스가 섞여 있다**는 것이다. 후자는
fixer가 없다. 문자열을 어디서 쪼갤지는 룰이 판단하지 않기 때문이다.

```
64:11  warning  'transition'의 카테고리가 올바르지 않습니다   tailwindcss-categorized/ordering
```

157건의 대부분이 후자였다. 그러니까 이 컨벤션의 핵심은 "클래스를 정렬해라"가 아니라
"**cn()의 문자열 인자 하나는 한 카테고리만 담아라**"다. 결국 손으로(정확히는
에이전트를 시켜) 전부 쪼갰다.

```tsx
// before
cn('flex items-center px-4 text-sm transition')
// after — 레이아웃 / 구조 / 스타일 / 트랜지션
cn('flex items-center', 'px-4', 'text-sm', 'transition')
```

한 가지 조심한 것: cn은 tailwind-merge를 쓰므로 충돌하는 유틸리티(p-2 vs p-4)가 같은
호출에 있으면 인자 순서가 최종 클래스를 바꾼다. 분리 전에 충돌 클래스 여부를 확인하게
했고, 사후 리뷰 표본 검사에서도 스타일이 바뀐 케이스는 없었다.

## 3. 서버 테스트의 any 지옥

서버 경고 111건 중 80건가량이 테스트 파일에서 났다. supertest의 응답 바디가 any라서
`res.body.accessToken`에 접근할 때마다 unsafe-member-access가 쌓인 것이다. 원본
템플릿(p2z)이 쓰는 패턴 그대로, 각 spec 상단에 응답 타입을 선언하고 캐스팅하는
방식으로 통일했다. 단언의 의미는 그대로 두고 타입만 입혔다.

나머지는 컨트롤러 public 메서드의 명시 반환 타입(17건), catch 후 재던질 때 원인을
잃지 않도록 `{ cause }` 부착(6건) 같은 것들. 전부 잡고 나서 base 설정의 type-safety
계열 룰을 일괄 error로 승격했다. 이제 어기면 빌드가 깨진다. 잔여 warning은 단
1건 — 숫자 입력 컴포넌트의 effect 내 setState인데, 외부 값과 입력 초안의 동기화
패턴이라 룰을 어기는 게 맞다고 보고 의도적으로 남겼다.

## 4. 561줄짜리 페이지 컴포넌트

p2z 컨벤션 위반을 전수 조사시켰더니 의외의 결과가 나왔다. 레이어 위반, 임포트 정책,
명명, 죽은 코드, i18n 키 — 전부 0건. 문제는 단 두 종류였다: 중복 코드 약간과
**거대 페이지 컴포넌트**. 노력치 역산 페이지가 561줄, 배틀 페이지가 408줄이었다.

분해 결과:

| 페이지 | before | after | 추출 |
|---|---|---|---|
| 노력치 역산 | 563줄 | 51줄 | 모드 3개 + 결과 행 + 공유 옵션 |
| 배틀 | 413줄 | 113줄 | 섹션 카드 6개 |
| 매치업 | 353줄 | 247줄 | 메가 폼 선택 섹션 |
| 매치업 매트릭스 | 321줄 | 193줄 | 팀 입력 카드 + 범례 |
| 파티빌더 | 238줄 | 134줄 | 슬롯 컴포넌트 |

스피드 페이지(237줄)는 일부러 안 쪼갰다. 이미 파일 안에서 로컬 컴포넌트 세 개로
응집돼 있어서, 파일만 늘리고 가독성은 안 늘 거라는 판단이었다.

조사 보고서가 제안한 것 중 기각한 항목도 있다. "여러 페이지가 유사한 옵션 배열을
중복 정의하니 공통 상수 모듈로 추출하라"는 제안이었는데, 실물을 보니 노력치 역산의
도구 배율 옵션과 배틀의 날씨·상태이상 목록은 **모양만 닮았지 내용이 전혀 다른**
배열이었다. 합치면 중복 제거가 아니라 과추상화다. 같은 이유로 세 모드에 반복되는
상태 갱신 헬퍼도 모드별 state 타입이 달라 제네릭으로 묶지 않고 로컬에 뒀다.
"최소한의 코드"라는 원칙은 추출을 명령할 때도 있지만 추출을 막을 때도 있다.

## 5. 리뷰어 패스가 잡은 것

"동작 변경 없는 순수 분리"라고 주장하는 커밋이야말로 리뷰가 필요하다. 테스트 33개로는
렌더 동작 드리프트를 다 못 잡으니, 별도 리뷰어에게 추출 전후 JSX 동등성·상태 흐름·
상수 무결성·핸들러 바인딩을 대조시켰다.

결과는 한 건. 매트릭스 페이지의 "포켓몬 추가" 핸들러에서 6마리 캡 가드가 분해 중에
사라졌다. 버튼 자체가 6마리째에 숨겨지므로 당장 버그는 아니지만, 원본에 있던 이중
방어 하나가 조용히 없어진 것이다. 함수형 업데이트 안에서 검사하는 형태로 복원했다.

```ts
setMyTeam((prev) => (prev.length < MAX_TEAM ? [...prev, makeEntry()] : prev));
```

수백 줄을 옮기는 기계적 작업에서 사람이든 AI든 이런 게 꼭 하나씩 샌다. "순수 분리"
주장은 검증 대상이지 전제가 아니다.

## 6. 함정 메모가 실제로 터졌다

이전 세션에 적어 둔 함정 메모가 있다 — "unused-imports 룰이 error+autofix라서, import
추가와 사용처 추가를 별개 편집으로 하면 그 사이에 포맷터가 import를 지운다." 이번
세션에서 그대로 재현됐다. 캡 가드를 복원하면서 import 문을 먼저 고치고 사용처를 뒤에
고쳤는데, 두 편집 사이에 저장 훅의 autofix가 끼어들어 "아직 안 쓰는" import를
삭제했다. 사용처 편집이 끝난 뒤 grep해 보니 import 줄만 증발해 있었다.

다행히 사용처가 이미 있는 상태에서 import를 다시 추가하면 안 지워진다. 같은 함정을
두 번 밟고 나서야 규칙이 체화됐다: **import와 사용처는 반드시 한 편집에 넣는다.**

## 7. 문서 빚 청산

코드만 아니라 문서도 한 차례 현행화했다. 서버 컨텍스트 문서는 "다음은 프리셋 티어"
라고 적혀 있었는데 실제로는 그 뒤로 쿼터·공유·전적·리더보드·스크린샷 조언까지 다
들어간 상태였다. README는 아직 서버가 Fastify라고 주장하고 있었고, 페이지 표는 6개
시절 그대로였다(현재 13개). 클립보드 paste 시절의 프롬프트 가이드 문서 3종은 코드로
진실이 옮겨진 지 오래라 삭제했다.

문서는 코드와 달리 컴파일러가 거짓말을 안 잡아 준다. 굵직한 단계가 끝날 때마다
이렇게 일괄 대조하는 수밖에 없는 것 같다.
