# 3편 — AI 분석 도입과 시행착오

웹앱이 도는 것을 확인한 뒤 본래 목표였던 AI 분석을 붙였다. "이 파티의 약점은", "이 상대 팀에
누구를 선출할까", "지금 이 상황에서 뭘 해야 하나" 같은 질문에 Claude가 답하게 만드는 일이다.
이 과정은 두 번의 방향 전환을 거쳤고, 둘 다 화면이 남아 있다.

## 첫 번째 시행착오: 클립보드 왕복 UX

초기 설계는 클립보드 paste 왕복이었다. 앱이 파티 정보를 직렬화해 클립보드에 담아주면,
사용자가 그걸 Claude 대화창에 붙여넣고, Claude의 답변을 다시 복사해 앱의 사이드패널에
붙여넣는 방식이었다. 아래가 그 시절의 실제 화면이다. 이 글을 쓰며 당시 커밋을 체크아웃해
다시 띄웠다.

어긋나도 파싱이 깨졌다. 스크린샷의 노력치 입력(252/252/4)이 보여주듯 이때는 아직 본가 EV

결국 이 플로우는 통째로 폐기했다.

## 전환: 서버가 Anthropic을 직접 호출

대신 서버가 Anthropic API를 직접 호출하도록 바꿨다. 프롬프트 본문은 도메인 계층의 직렬화
함수가 만든다 — 파티·배틀 상태를 받아 결정론 분석(상성·스피드·데미지) 결과까지 붙여 텍스트로
펼치는 함수다. 한카리아스 한 마리짜리 파티를 넣고 실제로 뽑아본 본문의 앞부분이다.

```
## 작업: 파티 분석

## 파티
1. 한카리아스 Lv50
   특성: 까칠한피부 / 도구: 구애스카프 / 성격: 고집 / 테라: 땅
   기술: 지진, 역린, 스톤에지, 불꽃엄니
   EV: H0/A32/B0/C0/D1/S32

## 정적 분석 (이미 계산된 값, 재계산 불필요)
- 약점 분산 점수: 0/100 (피크 1슬롯)
- 타입별 약점 슬롯 수:
  - 얼음: 1슬롯
  - 드래곤: 1슬롯
  - 페어리: 1슬롯
- 역할 분포: 고속어태커 1
- 화력 합계: 물리 200, 특수 90, 내구 404, 최고스피드 154

## 챔피언스 룰 메모
- 한 배틀에 메가진화는 1마리만 가능하다. ...

## 챔피언스 도구 카탈로그 (이 리스트 밖 도구 절대 추천 금지)
- 배리열매
- 구애스카프
- ...
```

설계 의도가 이 본문에 다 들어 있다. 상성·실수치 같은 **계산은 전부 우리 결정론 계층이 미리
해서 "이미 계산된 값, 재계산 불필요"로 못 박아 넘기고**, 모델에게는 그 위의 판단만 시킨다.
챔피언스에 존재하는 도구 카탈로그를 통째로 실어 "이 리스트 밖 도구 절대 추천 금지"로 환각의
공간 자체를 좁힌다.

응답은 느슨한 텍스트 파싱 대신 Structured Outputs로 스키마를 강제했다.

```ts
const response = await this.anthropic.messages.parse({
  model: MODEL_BY_TASK[task],
  max_tokens: 1500,
  system: [{ type: 'text', text: SYSTEM, cache_control: { type: 'ephemeral' } }],
  messages: [{ role: 'user', content: body }],
  output_config: { format: zodOutputFormat(ClaudeResponseSchema) },
});
```

모델은 작업별로 환경변수로 갈아끼울 수 있게 라우팅 테이블로 뒀다(기본은 전부 Sonnet).

```ts
// apps/server/src/advisor/advisor.service.ts
const MODEL_BY_TASK: Record<ExportTask, string> = {
  'party-analysis': process.env.ADVISOR_MODEL_PARTY ?? 'claude-sonnet-4-6',
  'matchup-leadrec': process.env.ADVISOR_MODEL_MATCHUP ?? 'claude-sonnet-4-6',
  'battle-decision': process.env.ADVISOR_MODEL_BATTLE ?? 'claude-sonnet-4-6',
};
```

응답 스키마(`ClaudeResponseSchema`)는 Zod로 정의했다. 요약·세부 항목·실행 가능한 제안에
더해, 마지막 두 필드가 다음 절의 복선이다.

```ts
// packages/pokedex-core/src/parse.ts
export const ClaudeResponseSchema = z.object({
  task: z.enum(['party-analysis', 'matchup-leadrec', 'battle-decision']),
  summary: z.string(),
  details: z.array(
    z.object({
      kind: z.enum(['strength', 'weakness', 'warning', 'recommendation']),
      target: z.string(),
      text: z.string(),
    }),
  ),
  unknownNames: z.array(z.string()).default([]),
  // 응답에 등장시킨 모든 포켓몬·기술·특성·도구 한국명. 서버가 검증 사전과 대조해 음역·fabricate를 잡는다.
  mentionedNames: z.array(z.string()).default([]),
});
```

정적 시스템 프롬프트에는 prompt caching을 걸어 연속 호출의 입력 토큰 비용을 줄였다.

## 두 번째 시행착오: 한국어 고유명사를 지어낸다

가장 골치 아픈 문제는 모델이 한국어 고유명사를 fabricate하는 것이었다. 처음에 비용을 아끼려
가벼운 모델을 썼더니, 포켓몬 종족명·기술명·특성명을 그럴듯하게 지어내는 사례가 잦았다.
실제로 존재하지 않는 이름을 자신 있게 출력하는 것이라, 분석 결과 자체를 신뢰할 수 없게 만든다.

두 갈래로 대응했다.

첫째, 추천 계열 모델을 Sonnet으로 고정했다. 한국 SV 커뮤니티 어휘 정확도가 확연히 달랐다.
(이미지 OCR만은 정확도 때문에 별도로 Opus 비전을 쓴다.)

둘째, 응답에 2패스 명칭 보정을 넣었다. 모델이 응답에서 언급한 모든 고유명사를
`mentionedNames`로 함께 반환하게 하고, 이를 검증 사전과 대조한다. 사전에 없는 이름이
있으면 그 목록을 다시 모델에 주고 한 번 더 교정을 요청한다. 실제 코드는 이렇다.

```ts
// apps/server/src/advisor/advisor.service.ts
let unverified = parsed.mentionedNames.filter((name) => !isKnownTerm(name));
if (unverified.length > 0) {
  const correction = `\n\n## 명칭 교정 요청 (필수)\n다음 명칭은 검증 사전에 없다(음역·직역·오타·미존재 의심): ${unverified.join(', ')}\n정식 한국 명칭으로 바꾸거나, 확신이 없으면 해당 언급을 통째로 삭제하고 응답을 다시 작성하라. mentionedNames도 갱신하라.`;
  const retried = await this.callClaude(task, body + correction);
  const retriedUnverified = retried.mentionedNames.filter((name) => !isKnownTerm(name));
  if (retriedUnverified.length < unverified.length) {
    // 재시도가 더 깨끗하면 채택
  }
}
```

`isKnownTerm`은 PokeAPI에서 수집한 종족·기술·특성·도구 사전을 합쳐 만든 검증 함수다.
끝까지 검증을 통과 못 한 이름은 응답에서 "미확인 명칭"으로 분리해 사용자에게 그대로
노출하지 않는다.

시스템 프롬프트에도 못을 박았다. 실제 프롬프트에서 발췌한 규칙 세 줄이다 — 전부 실패
사례를 겪고 나서 추가된 것들이라, 한 줄 한 줄이 사건 기록이기도 하다.

```
영어 음역·직역 절대 금지. 반드시 한국 정식 명칭만.
예: '샌드스트림'→'모래날림', '로키헬멧'→'울퉁불퉁멧', '드래곤테일'→'용의꼬리',
'돌머리 계열'·'돌진계' 같은 애매한 총칭 금지.

상대 포켓몬의 특성·기술은 입력에 없으면 모르는 정보다. 추측해서 단정하지 말 것.
모르는 특성을 지어내느니 '상대 특성 불명'으로 두는 게 낫다.

포켓몬을 가리킬 때는 입력에 적힌 한국어 종족명을 그대로 쓴다.
'슬롯 1번', '1번 슬롯' 같은 슬롯 번호 표현 절대 금지.
```

한국 커뮤니티에서 통용되는 표현으로만 답하게 만드는 것이 핵심이었다. "샌드스트림"은 영어
음역이고 한국 정발 명칭은 "모래날림"이다 — 이런 쌍을 모델이 알아서 맞춰주리라 기대하면
안 되고, 예시까지 박아서 가르쳐야 했다.

## 결과로 만들어진 AI 기능

이렇게 네 가지 AI 기능이 자리잡았다. 파티 정적 분석, 매치업 선출 추천, 실시간 배틀 조언은
Sonnet으로, 파티 화면 이미지에서 파티를 읽어내는 import는 Opus 비전으로 동작한다. 각 추천은
결정론 분석(상성·스피드·데미지) 결과를 프롬프트에 함께 실어, 모델이 근거 있는 답을 하도록
유도했다. (다섯 번째 AI 기능인 배틀 스크린샷 조언은 7편에서 다룬다.)

실제로 리자몽+마기라스 2마리 파티를 분석시킨 화면이다. 요약 한 줄("물 2약점·파티 2마리
한계, 특수 화력 우위지만 내구 부족")과 포켓몬별 장점·단점이 구조화 응답 그대로 렌더링된다.
"가속 특성으로 고스피드 유지", "모래날림+구애스카프로 물리 297 화력" — 화력 수치는
프롬프트에 실어준 결정론 계산값을 모델이 그대로 근거로 쓴 것이다.

## 정리

AI를 붙이는 일은 "API를 호출한다"가 전부가 아니었다. 어떤 UX로 데이터를 주고받을지(왕복
폐기), 어떤 모델이 한국어 도메인에 맞는지(Sonnet), 그리고 모델이 틀린 이름을 낼 때 어떻게
잡을지(2패스 보정)가 실제 일의 대부분이었다. 다음 편은 이 모든 걸 떠받칠 백엔드를 NestJS로
다시 세운 이야기다.
